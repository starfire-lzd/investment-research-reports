# AI 驱动投资体系

本文件定义每日自动化投研闭环。目标不是生成更多日报，而是让盘前假设、盘中验证、尾盘决策、盘后复盘和长期规则沉淀形成同一套可追踪系统。

## 总体原则

- 先判断是否为 A 股交易日；非交易日不创建文件、不发布、不发送。
- 所有分析先读取 `rules.md`、`watchlist.md`、`memory.md`、`positions/当前持仓.md`，再结合当天事实。
- 所有交易建议必须受 `risk.md` 约束；风险约束与方向判断冲突时，以风险约束为准。
- 所有建议必须给出触发条件、失效条件、仓位边界和风险等级。
- 交易建议优先来自 `watchlist.md`；临时新增标的必须说明为什么值得脱离观察池观察。
- 盘后只把跨日期可复用的经验写入 `memory.md`；短期行情、今日热点和下一交易日方向留在日报中。
- 判断条件的成功度写入 `判断条件评分.md`，不要混入当天行情流水。
- `rules.md` 和 `watchlist.md` 默认只在复盘中提出建议，不自动修改，除非明确要求执行调整。
- 复盘中提出的新规则先进入 `rules_proposals.md`，连续验证后再升级到 `rules.md`。

## 每日节奏

| 时间 | 自动化 | 主要产物 | 核心职责 |
| --- | --- | --- | --- |
| 09:00 | 开市前准备 | `YYYY-MM-DD/盘前分析.md` | 建立今日假设、风险边界、观察优先级 |
| 10:00 | 盘中分析 | `YYYY-MM-DD/盘中分析.md`、`plans/YYYY-MM-DD_交易计划.md` | 用开盘 30-60 分钟事实验证盘前假设，形成可执行交易计划 |
| 14:25 | 尾盘分析 | `YYYY-MM-DD/尾盘分析.md` | 判断尾盘是否确认、修复或证伪早盘方向，给出收盘前处理建议 |
| 19:00 | 今日复盘 | `YYYY-MM-DD/盘后复盘.md`、`trades/YYYY-MM-DD_交易日志.md`、`data/YYYY-MM-DD/signal-targets.json`、`data/YYYY-MM-DD/signal-evaluation.json`、`判断条件评分.md`、必要时更新 `memory.md` | 验证全天判断，回看三交易日前判断，沉淀经验、教训和可复用策略 |

## 盘前分析

盘前任务回答三个问题：

1. 昨日主线今天是否仍有延续基础。
2. 哪些 `rules.md` 规则可能触发，哪些方向必须降低预期。
3. 当前持仓与今日可能主线是否匹配，风险在哪里。

输出必须包含：

- 今日市场情绪判断。
- 上一交易日主线延续性判断。
- 今日可能触发的交易规则。
- `watchlist.md` 观察池验证重点。
- 当前持仓风险与机会。
- 今日重点投资方向、候选标的或 ETF、应回避方向。
- 盘中需要重点观察的触发条件和失效条件。
- 今日风险预算：总仓位上限、单方向上限、单票上限、禁止事项。

## 盘中分析

盘中任务把盘前假设转换成可执行计划。重点不是追热点，而是确认哪些方向通过了开盘 30-60 分钟验证。

输出必须包含：

- 盘前观点验证：哪些被验证，哪些被证伪。
- `rules.md` 规则触发情况。
- `watchlist.md` 观察池强弱验证。
- 当前持仓是否需要减仓、持有、观察或禁止加仓。
- 今日优先投资方向和个股或 ETF 观察池。
- 观察建议、条件交易建议、风险处理建议三类必须分开。
- `plans/YYYY-MM-DD_交易计划.md`，每个计划必须有买入触发、无效条件、止损条件、仓位上限和计划类型。

## 尾盘分析

尾盘任务在 14:25 执行，只解决收盘前决策问题，不重复早盘报告。

输出必须包含：

- 早盘主线在午后是否延续、分歧、修复或证伪。
- 交易计划中哪些标的已经触发、未触发、接近触发或被证伪。
- 当前持仓尾盘处理建议：持有、减仓、止损、禁止加仓、隔夜观察。
- 是否允许尾盘开新仓；若允许，必须说明仓位上限和隔夜风险。
- 收盘后复盘必须验证的问题。

尾盘建议应更偏风控：

- 核心股尾盘回落且不能收回关键分时位置时，降低隔夜风险。
- 后排午后冲高但容量核心不跟随时，不作为买入信号。
- 尾盘只允许处理已经在交易计划内、且条件明确触发的标的。

## 今日复盘

复盘任务构成系统学习环节。它必须区分事实、判断、执行和沉淀。

输出必须包含：

- 今日市场总体复盘。
- 盘前策略复盘。
- 盘中策略复盘。
- 尾盘策略复盘。
- 交易计划执行/验证复盘。
- `rules.md` 规则执行复盘。
- `watchlist.md` 观察池表现复盘。
- 当前持仓表现与组合风险复盘。
- 今日主线和资金流向。
- 个股或 ETF 表现复盘。
- 错过机会复盘：今日强但系统未覆盖的方向、缺失原因、是否进入观察池候选。
- 写入 `memory.md` 的可复用经验；如果没有写入，说明未写入原因。
- 三交易日前判断回看与条件成功度更新。
- `rules.md` / `watchlist.md` 变更建议。
- 下一交易日方向排序和重点观察信号。

如果没有真实交易记录，必须写明 `未提供真实交易记录`，不得编造买卖价格、成交、仓位或盈亏。

## 三交易日前回看

今日复盘必须额外回看三交易日前的分析和交易计划。目标是检验当时的判断在三交易日后的表现，而不是事后重写逻辑。

复盘前必须先由模型完成标的抽取，再运行脚本。脚本不从 Markdown 文本里猜测标的。

模型需要先读取 T-3 的分析和交易计划，创建 `data/YYYY-MM-DD/signal-targets.json`：

```json
{
  "targetDate": "YYYY-MM-DD",
  "t3Date": "YYYY-MM-DD",
  "sources": [
    "YYYY-MM-DD/盘前分析.md",
    "YYYY-MM-DD/盘中分析.md",
    "YYYY-MM-DD/尾盘分析.md",
    "plans/YYYY-MM-DD_交易计划.md"
  ],
  "securities": [
    {
      "code": "000000",
      "name": "标的名称",
      "reason": "原始判断摘要",
      "source": "来自盘前/盘中/尾盘/交易计划",
      "conditionIds": []
    }
  ]
}
```

然后运行：

```bash
npm run evaluate:signals
```

脚本产物为 `data/YYYY-MM-DD/signal-evaluation.json`。复盘正文以该 JSON 为价格验证依据；如果 `signal-targets.json` 缺失或脚本失败，需要说明失败原因，再降级为人工可得证据，不得编造价格。

输入文件：

- `T-3/盘前分析.md`
- `T-3/盘中分析.md`
- `T-3/尾盘分析.md`
- `plans/T-3_交易计划.md`
- `T-3/盘后复盘.md`

如果某个文件不存在，需要明确说明缺失，并使用可获得文件继续评估。

评估维度：

- 当时看多或优先观察的方向，在三交易日后是否跑赢主要指数或同类 ETF。
- 当时列出的个股或 ETF，当前股价相对触发日收盘、计划触发价或可获得基准价的涨跌幅。
- 当时设置的触发条件、失效条件和风险提示是否有效。
- 判断准确时，记录哪些条件贡献最大，并提高对应条件的成功度。
- 判断不准确时，分析原因：逻辑错误、节奏错误、资金不认可、指数环境变化、突发消息、个股基本面或走势不匹配。

输出必须包含：

- `T-3` 日期和读取到的文件清单。
- 每个方向或标的的原始判断摘要。
- 当前股价或可获得价格表现。
- 判断结果：准确 / 部分准确 / 不准确 / 无法验证。
- 成功或失败原因。
- 对 `判断条件评分.md` 的更新内容。

`判断条件评分.md` 更新规则：

- 条件被验证有效时，增加样本数和成功次数，并记录最新证据。
- 条件被证伪时，增加样本数和失败次数，并记录失败原因。
- 条件只是无法验证时，不更新成功率，只记录缺失原因。
- 不因单日结果直接改写 `rules.md`，除非用户明确要求；只在复盘里提出规则调整建议。

## 风险约束

每日计划必须读取 `risk.md`，并把建议分为三类：

- 观察建议：只进入观察池，不触发交易。
- 条件交易建议：只有满足交易计划中的触发条件、失效条件、仓位上限和止损条件时才允许执行。
- 风险处理建议：减仓、止损、禁止加仓、降低隔夜暴露。

任何方向判断不能覆盖风险约束。若 `risk.md` 限制仓位或禁止追高，日报必须明确写出约束原因。

## 错误统计

复盘中判断不准确时，必须归类错误类型：

- 逻辑错误。
- 节奏错误。
- 资金不认可。
- 指数环境变化。
- 后排误判。
- 数据缺失。
- 突发消息。
- 个股基本面或走势不匹配。

如果同一错误类型连续出现，需要进入 `rules_proposals.md`，但不直接改写 `rules.md`。

## 沉淀机制

`memory.md` 只接受以下内容：

- 可跨日期复用的判断模型。
- 已验证有效的市场信号。
- 已验证失效的判断模式。
- 风控规则和流程规则。

`判断条件评分.md` 接受以下内容：

- 条件名称。
- 适用场景。
- 样本数、成功次数、失败次数、成功率。
- 最近一次验证日期、验证结果和证据摘要。
- 失败模式或需要继续观察的问题。

`rules.md` 的新增或调整建议需要满足至少一个条件：

- 同类问题连续出现。
- 某条规则在当天明确降低亏损或过滤错误机会。
- 某条规则被事实证伪，需要改变触发条件或风控条件。

`watchlist.md` 的新增方向需要满足至少一个条件：

- 连续两个交易日获得资金认可。
- 出现容量中军和前排弹性标的共同走强。
- 盘后复盘证明该方向不是单日消息刺激。
